# デプロイ・公開ガイド 🚀

「つづく」を他の人が使えるようにインターネットへ公開する手順です。
エンジニア学習中の方向けに、考え方（なぜそうするか）も添えています。

---

## 0. まず前提：このアプリは「静的サイト」

「つづく」は、すべての処理がブラウザの中で完結し、データも各自のブラウザの
localStorage に保存されます。**サーバー側で動くプログラムやデータベースは使っていません。**

こういうアプリを「静的サイト（static site）」と呼びます。公開するために必要なのは、
`npm run build` で作られる `dist/` フォルダ（HTML・CSS・JS の完成品）を、
**ファイルを置くだけのホスティング（静的ホスティング）にアップロードする**、それだけです。

> ⚠️ 重要な性質：データは「その人のブラウザ」にしか保存されません。
> - ユーザーAとユーザーBでデータは共有されません（各自の記録は各自の端末だけ）
> - 同じ人でも、PCとスマホでは別々の記録になります
> - これは個人用トラッカーとしては自然な挙動です。
>   もし「ログインして、どの端末でも同じデータを見たい」を実現したい場合は、
>   サーバー・データベース・認証（バックエンド）が必要になり、規模が一段上がります。
>   → 本ガイドの最後「次のステップ」を参照。

---

## 1. 本番ビルドを作る

公開用の完成ファイルを生成します。プロジェクトフォルダで:

```
npm run build
```

`dist/` フォルダができます。中身は `index.html` と `assets/` の中の JS・CSS だけ。
これが「出荷する箱」です。手元で見た目を確認したいときは:

```
npm run preview
```

`dev` は開発用（毎回ソースを変換）、`build`＋`preview` は本番と同じ最適化済みの状態、
という違いがあります。

## 2. 公開する（3つの選択肢）

かんたんな順に3つ紹介します。**まずは方法Aがおすすめ**です。

### 方法A：Netlify Drop —— 最速・アカウントほぼ不要

`dist` フォルダをWebページにドラッグするだけでURLが発行されます。

1. `npm run build` を実行して `dist/` を作る
2. ブラウザで **https://app.netlify.com/drop** を開く
3. エクスプローラーから **`dist` フォルダをページ上にドラッグ＆ドロップ**
4. 数秒で `https://ランダムな名前.netlify.app` のような公開URLが発行される
5. そのURLを共有すれば、誰でもアクセスできます

- 更新したいとき：また `npm run build` して、同じ場所にドラッグし直すだけ
- 無料。独自ドメインやサイト名の変更は、無料アカウント登録で可能

### 方法B：Vercel ＋ GitHub —— 実務に近い自動デプロイ

コードを GitHub に置き、Vercel と連携させると、**コードを push するたびに自動で
ビルド＆公開**されます（CI/CD の入り口）。少し準備は要りますが、最も「実務的」です。

1. コードを Git 管理して GitHub にリポジトリを作成・push する
   （`node_modules` と `dist` は `.gitignore` で除外済みなので上げなくてOK）
2. **https://vercel.com** に GitHub アカウントでサインイン
3. 「Add New… → Project」で該当リポジトリを選ぶ
4. Vercel が Vite を自動検出。設定はほぼ既定のままでOK
   - Build Command: `npm run build`
   - Output Directory: `dist`
5. 「Deploy」を押すと公開URLが発行される
6. 以降は `git push` するだけで自動的に再ビルド・再公開される

### 方法C：GitHub Pages —— GitHubだけで完結・無料

すでに GitHub を使うなら、追加サービスなしで公開できます。

1. コードを GitHub リポジトリに push
2. 手動なら：`npm run build` 後、`dist/` の中身を `gh-pages` ブランチに公開する
   （`gh-pages` パッケージや GitHub Actions を使うのが定番）
3. リポジトリの Settings → Pages で公開ブランチ／フォルダを指定
4. `https://ユーザー名.github.io/リポジトリ名/` で公開される

> 💡 このプロジェクトは `vite.config.js` で `base: './'`（相対パス）に設定済みなので、
> `github.io/リポジトリ名/` のようなサブパス配信でも、CSSやJSが正しく読み込まれます。
> これはハマりやすいポイントなので、あらかじめ対応してあります。

## 3. 公開後のチェックリスト

- [ ] 発行されたURLを、スマホなど別の端末でも開いて表示を確認
- [ ] 習慣の追加・チェック・年間記録が動くか確認
- [ ] （任意）独自ドメインを設定する（各サービスの管理画面から）
- [ ] （任意）`<title>` やファビコン、OGP画像など「見た目のメタ情報」を整える

## 4. どれを選ぶ？

| | 手軽さ | 自動更新 | 向いている人 |
|---|---|---|---|
| A. Netlify Drop | ◎ 最速 | ×（毎回ドラッグ） | まず世に出したい |
| B. Vercel + GitHub | ○ | ◎ | 継続開発・実務練習 |
| C. GitHub Pages | △ | ○（要設定） | GitHubで完結したい |

学習目的なら、**まずAで「公開できた」体験をしてから、Bに進んでGit連携と
自動デプロイを体験する**のがおすすめの順番です。

---

## 5. 次のステップ（本格的なプロダクトにするなら）

今は「各端末にデータが残る個人アプリ」です。ここから先に進めたくなったら:

- **アカウント＆データ同期**：Supabase や Firebase を使うと、認証とデータベースを
  比較的少ない手間で追加でき、「ログインすればどの端末でも同じ記録」が実現できます。
- **PWA化**：マニフェストと Service Worker を足すと、スマホのホーム画面に
  アプリのように追加でき、オフラインでも起動できます。
- **通知・リマインダー**：Web Push を使えば「今日の習慣まだですよ」を送れます。

これらは静的サイトから一歩踏み込む内容なので、やりたくなったら一緒に設計しましょう。
